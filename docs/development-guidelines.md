# 開発ガイドライン

## 1. コーディング規約

### 1.1 共通
- 1つの関数・コンポーネントは1つの責務に集中させる
- 早期リターンを活用し、ネストを深くしない
- マジックナンバー・マジック文字列は定数化する
- コメントは「なぜ」を説明する場合のみ記載し、「何をしているか」は書かない（コードで表現する）

### 1.2 バックエンド（FastAPI / Python）
- フォーマット・Lintは Ruff に従う（`ruff format`, `ruff check`）
- 型ヒントを必須とする（関数の引数・戻り値）
- リクエスト/レスポンスは必ず Pydantic スキーマで定義し、辞書を直接返却しない
- DBアクセスは `services/` 層に集約し、ルーターでは直接ORMクエリを書かない
- 例外はFastAPIの `HTTPException` で適切なステータスコードを返す
- 環境変数・設定値は `core/config.py` に集約し、コード中に直接埋め込まない

### 1.3 フロントエンド（React / TypeScript）
- フォーマット・Lintは ESLint + Prettier に従う
- コンポーネントは関数コンポーネント + Hooksで実装する
- `any` 型の使用は避け、必要な場合はその理由をコメントで明記する
- API通信は TanStack Query を介して行い、コンポーネント内で直接 `fetch` しない
- フォームの入力値検証は Zod スキーマで定義し、バックエンドのPydanticスキーマと項目を一致させる
- グローバルなUI状態（認証情報等）以外は、ローカルstateまたはサーバー状態（TanStack Query）で管理し、不要なグローバルステートを増やさない

## 2. 命名規則

### 2.1 共通
- 識別子は英語表記とし、ドメイン用語は `docs/glossary.md` の英語表記に統一する

### 2.2 バックエンド
| 対象 | 規則 | 例 |
|---|---|---|
| ファイル名 | snake_case | `book_service.py` |
| クラス名 | PascalCase | `BookService`, `BookCreate` |
| 関数・変数名 | snake_case | `get_book_by_id` |
| 定数 | UPPER_SNAKE_CASE | `DEFAULT_PAGE_SIZE` |
| DBテーブル名 | snake_case（複数形） | `books`, `book_tags` |
| DBカラム名 | snake_case | `created_at` |
| APIパス | kebab-case（複数形リソース） | `/api/books`, `/api/tags` |

### 2.3 フロントエンド
| 対象 | 規則 | 例 |
|---|---|---|
| コンポーネントファイル名 | PascalCase | `BookTable.tsx` |
| その他ファイル名（hooks, utils等） | camelCase | `useBooks.ts` |
| コンポーネント名 | PascalCase | `BookTable` |
| 関数・変数名 | camelCase | `fetchBooks` |
| 型・インターフェース名 | PascalCase | `Book`, `BookStatus` |
| 定数 | UPPER_SNAKE_CASE | `DEFAULT_PAGE_SIZE` |
| カスタムフック | `use` + camelCase | `useBooks`, `useTags` |

## 3. スタイリング規約

- スタイリングは Tailwind CSS のユーティリティクラスを使用する
- 共通で繰り返し使うスタイルはコンポーネント化し、クラス文字列の重複を避ける
- 配色・余白等のデザイントークンは Tailwind の設定（`tailwind.config.js`）で管理し、コンポーネント側で独自のカスタム値（任意値）の多用は避ける
- レスポンシブ対応はTailwindのブレークポイント（`sm:`, `md:`, `lg:`）を用いる

## 4. テスト規約

### 4.1 バックエンド
- テストフレームワークは pytest を使用する
- APIエンドポイントのテストは `tests/api/` に配置し、httpxの非同期クライアントでリクエスト/レスポンスを検証する
- サービス層のテストは `tests/services/` に配置する
- 認証・認可（他ユーザーのデータにアクセスできないこと）は必ずテストする
- テストごとにDBを初期化し、テスト間で状態が引き継がれないようにする

### 4.2 フロントエンド
- テストフレームワークは Vitest + React Testing Library を使用する
- コンポーネントのテストは対象コンポーネントと同じディレクトリに配置する
- ユーザー操作（入力・クリック）を起点とした振る舞いベースのテストを優先し、実装の詳細に依存したテストは避ける

## 5. Git規約

### 5.1 コミットメッセージ
- 1コミット1関心事を基本とする
- コミットメッセージは「何を」「なぜ」変更したかが分かるように記載する

### 5.2 ブランチ運用
- `main` ブランチを安定ブランチとする
- 機能追加・修正は作業用ブランチを作成して行い、`main` への直接コミットは避ける

### 5.3 プルリクエスト
- 変更内容に対応する `.steering/` のドキュメントを参照できるようにする
- レビュー前にリント・型チェック・テストを実行し、すべて成功していることを確認する
